# 框架确定性审查报告

> 审查日期：2026-06-25 ｜ 审查范围：`Source/{core,hybrid,hal,algorithm,component,extensions}`
> 目的：核对框架各模块是否满足**确定性（determinism）**，输出可据以修复的问题清单。
> 后续：本报告中的 `DET-xx` 条目作为修复工单的唯一标识。

---

## 1. 审查方法与判据

嵌入式语境下"确定性"= **行为可复现 + 执行时间可静态上界（WCET 可分析）**。本次按以下判据逐层核对：

| 判据 | 含义 |
|---|---|
| D1 无运行期动态分配 | 无 `malloc/calloc/realloc/free/alloca`，仅用静态/池式内存 |
| D2 循环有界 | 无 `while(1)`/`for(;;)` 死循环；所有循环 trip count 有编译期或已校验的运行期上界 |
| D3 无无界自旋/阻塞 | 满/空走非阻塞返回（`WOULD_BLOCK`/`BUSY`），无忙等让步、无不封顶的 seqlock 重试 |
| D4 无递归 | 调用栈深度静态有界 |
| D5 临界区短而定长 | 关中断时长有界，无在临界区内做不定长操作 |
| D6 配置界已校验 | 运行期循环界来自配置时，init 期须对编译期 MAX 校验 |

审查手段：全 `Source/` 模式扫描（分配、循环、自旋、阻塞、CAS）+ 运行时核心逐函数精读 + 各层抽样确认。

---

## 2. 总体结论

**运行时核心（`core` + `hybrid`）确定性合格，且为有意识设计**——非偶然达成。证据：

- **D1 ✅ 全框架零动态分配**。`Source/` 无任何堆调用；`bm_mempool` 的 "free" 为池内定槽位图回收，O(1)。
- **D2/D3 ✅ 处理按预算封顶**。`bm_exec_drain_stream_slots(budget)`（`bm_exec.c:304`）、`bm_event_process(max_events)`（`bm_event.c:874`）均按上限参数迭代；`bm_exec.c:296` 注释明确"扫描开销计入 WCET 预算，使 WCET 可静态分析"。
- **D3 ✅ 非阻塞语义贯穿全线**。`bm_bus`/`bm_stream`/`bm_ultra`/`bm_sync` 满/空一律返回 `BM_ERR_WOULD_BLOCK`/`BM_ERR_BUSY`。
- **D3 ✅ 有界 CAS**。`bm_atomic_inc` 多核路径 CAS 环用 `BM_CONFIG_ATOMIC_MAX_RETRIES` 封顶后饱和（`bm_critical.c:78`，注释"避免无限自旋拖死 ISR/主循环"）。
- **拓扑运行前冻结**。`bm_event.c:887` 未冻结订阅表拒绝分发，冻结后链表不可变 → 分发遍历界 = 静态订阅数；`bm_bus`/`bm_module` 同样 freeze-then-run。
- **D5 ✅ 临界区 = 关中断**，定长操作，无嵌套膨胀（`bm_critical.c` 默认路径）。
- **D6 ✅ 配置界已校验**。`audio_array_frontend.c:89` 校验 `num_channels > BM_AUDIO_ARRAY_MAX_CHANNELS` 拒绝、`motion_coordination.c:30` 校验 `axis_count > BM_MOTION_COORD_MAX_AXES` 拒绝、`tinyml_adapter.c` 校验 arena ≤ `BM_TINYML_ARENA_MAX_BYTES`、`ndim≤4`、`SOFTMAX_MAX=16`。

## 3. 问题清单（修复工单）

### DET-01 ｜ 中 ｜ `bm_bus` LATEST 模式 seqlock 无界自旋 ｜ ✅ 已实现（待在可构建副本验证）

> **修复记录（2026-06-25）**：
> - `bm_bus.c` LATEST 读循环 `do/while` → `for(;;)` + `retry` 计数，超 `BM_CONFIG_BUS_LATEST_MAX_RETRIES`（默认 8）非阻塞返回 `BM_ERR_WOULD_BLOCK` 并清 `latest_reading`。
> - `bm_config.h` 新增 `BM_CONFIG_BUS_LATEST_MAX_RETRIES`（含 `< 1` 编译期 `#error`），与 `BM_CONFIG_ATOMIC_MAX_RETRIES` 同源。
> - 新增 `BM_ENABLE_BUS_TEST_HOOK` 测试缝（仿 `BM_ENABLE_EVENT_TEST_HOOK`）：`bm_bus.h` 声明、`bm_bus.c` 定义、CMake 为 `bm_core` 与 `test_bus` 注入宏。
> - `test_bus.c` 新增 `test_latest_read_retry_bounded_under_contention`：注入持续抢占钩子，断言返回 `WOULD_BLOCK`、钩子恰好调用 MAX 次、`latest_reading` 清回 `NONE`。
> - **验证（待办，本机工具链不全）**：在 Hoverboard 副本 `cmake -DBM_BUILD_TESTS=ON … && ctest -R test_bus`。修复前该用例会挂死（无界自旋），修复后应通过。


- **位置**：`Source/core/bm_bus.c:536-545`
- **现象**：
  ```c
  do {
      p = bus_load_cur(&st->latest_published);
      if (p == BM_BUS_LATEST_NONE) { ...; return BM_ERR_WOULD_BLOCK; }
      bus_store_cur(&st->latest_reading, p);
  } while (bus_load_cur(&st->latest_published) != p);
  ```
  经典 seqlock 读：写者在读窗口（两 load + 一 store）内发布即重试。
- **影响**：这是**运行时核心唯一 WCET 不封顶的点**。SPSC 单读者契约下实际几乎不命中、有界于写者发布频率；但形式上写者持续抢占可致读者饿死，破坏 WCET 可分析性（违 D3）。
- **同框架内不一致**：`bm_critical.c:78` 的 CAS 环已加 `MAX_RETRIES` 上界，本处未加。
- **建议修复**：给 LATEST 读加重试上界（复用/新增 `BM_CONFIG_BUS_LATEST_MAX_RETRIES`），超限返回 `BM_ERR_WOULD_BLOCK`（让调用方下一 tick 重取），与"非阻塞 + WCET 可静态分析"风格对齐。
- **波及面**：仅此一处 `.c` 实现（`bm_snapshot` 为设计文档、`bm_sync.c` 用不同实现，均不含同类自旋）。

### DET-02 ｜ 低（算法层）｜ 连通域标注数据依赖迭代

- **位置**：`Source/algorithm/bm_algo_image.c:177-213`
- **现象**：`do { 全图扫描 } while(changed)`，重复扫描法，最坏 O((W·H)²)。
- **影响**：**会终止**（像素数封顶，违 D2 仅在"次数数据依赖"意义上），但迭代次数依赖图像内容，WCET 高且不定。代码注释已自认"对大图复杂度偏高，适合小分辨率"。属算法层尽力而为函数，非运行时核心。
- **建议修复**：实现保持不变；在头文件契约标注**"非定时 / best-effort，禁止接入 block/frame 等带 deadline 的 exec 槽"**。归入 DET-03 的统一标注。

### DET-03 ｜ 低（文档/契约）｜ 缺少统一的 WCET 分类标注

- **位置**：`algorithm` 层数据依赖函数、`component/tinyml_adapter.c` 模型依赖推理。
- **现象**：这些函数 WCET = 数据/模型规模的函数（tinyml 受 arena 上限封顶但每模型需离线实测周期数）。调用者无法从签名判断能否进硬实时槽。
- **影响**：误接入定时槽会导致 deadline miss，是确定性"使用面"的隐患，而非实现缺陷。
- **建议修复**：在 `docs/05-API参考` 与各头文件为函数加 WCET 分类标签：
  - `O(1) 定时`：常数时间，可进任意实时槽；
  - `O(n) 配置界`：受已校验配置上限封顶，可进实时槽（需按上限预算）；
  - `best-effort`：数据/模型依赖，仅限非定时路径。

## IPC 统一重构确定性增补（2026-06-27）

> 审查范围：`include/bm/mp/bm_mp_ipc.h`、`include/bm/mp/bm_mp_ipc_backend.h`、`Source/mp/bm_mp_ipc_backend.c`
> 对应提交：`feat(ipc): add NxN matrix payload channels with bm_ipc_backend adapter`

### 阶段 1：`BM_BUS_IPC` 模式分流（bus 五入口）

`bm_bus` 的 `acquire_write` / `commit` / `abort` / `acquire_read` / `release` 五个入口增加 `mode == BM_BUS_IPC` 分流路径，逻辑为：

1. 判 vtable 指针是否为 NULL（无动态查找，无内存分配）。
2. 若非 NULL，直接委托后端 vtable 一跳（单次函数指针调用）。
3. 若 NULL，返回 `BM_ERR_INVALID`。

**D1 合格**：仅静态 vtable 指针跳转，无 `malloc`/`alloca`/池分配。
**D2 合格**：无循环，分流路径为直线流。
**D3 合格**：无自旋，无阻塞，不合法时立即返回错误码。

---

### 阶段 2：matrix payload 通道 + adapter

#### 2-A FIFO 命令环 `cmd_ring[N][N]`

- 数据结构：每对（写核, 读核）独占一个定长槽数组，`head` / `tail` 为绝对游标（`uint32_t`，自然回绕），槽数 `BM_CONFIG_MP_IPC_CMD_RING_DEPTH` 为编译期常量。
- 可见性：写端在填槽后执行 **release fence**，读端在取槽前执行 **acquire fence**，确保跨核顺序可见；环形 SPSC 结构下单屏障对足够，无需每槽 CRC。
- 满/空语义：
  - 写端满时：**非阻塞**返回 `BM_ERR_OVERFLOW`，不等待消费方。
  - 读端空时：**非阻塞**返回 `BM_ERR_WOULD_BLOCK`，不自旋。

**D2 合格**：无等待循环，满/空立即返回。
**D3 合格**：无自旋，满/空均为非阻塞返回。

#### 2-B LATEST 遥测通道 `tel_channel[N][N]`

- 数据结构：每对（写核, 读核）独占一个单槽遥测缓冲区，使用 seqlock 保护并附 `CRC32` 防数据撕裂。
- 写端：递增序号（奇数 = 写中），填写 payload，计算 `bm_crc32`，再递增序号（偶数 = 稳定），执行 release fence。
- 读端：
  1. 读序号；若为奇数（写中），**立即返回** `BM_ERR_WOULD_BLOCK`，**不自旋**。
  2. 读 payload + CRC，执行 acquire fence。
  3. 再读序号；若序号变化（写者插入），**立即返回** `BM_ERR_WOULD_BLOCK`，**不自旋**。
  4. 校验 CRC32；不符则返回 `BM_ERR_CHECKSUM`。
  5. 全部通过，返回成功。

**D3 合格**：读端检测到不稳定或序号变化时立即返回，**无不封顶的 seqlock 重试循环**，调用方自行在下一 tick 重取。

#### 2-C payload × CRC 成本说明

- `bm_crc32` 采用**逐字节逐位实现**（非查表），吞吐 ≈ 8 cycle/byte（依目标 MCU 流水线）。
- WCET 表达式：`T_crc ≈ 8 × payload_len` cycle。
- payload 尺寸由编译期常量 `BM_CONFIG_MP_IPC_CMD_PAYLOAD_SIZE`（FIFO 侧）和 `BM_CONFIG_MP_IPC_TEL_PAYLOAD_SIZE`（LATEST 侧）决定，**有静态上界，可纳入 WCET 预算**。

**D2 合格**：CRC 循环 trip count = `payload_len` = 编译期常量，有界。

#### 2-D adapter 热路径（`bm_mp_ipc_backend`）

- `send` / `recv` 热路径仅含：
  - `memcpy(..., ctx->elem, payload_size)`（定长，payload_size 为编译期常量）。
  - FIFO/LATEST 通道读写各一次（见 2-A / 2-B）。
  - 无动态分配，无辅助循环。
- adapter 自包含：**不引用 `matrix.c` 多核 HAL**，所有多核相关操作（fence、seqlock、CRC）内联于 `bm_mp_ipc_backend.c`。

**D1 合格**：无动态分配。
**D2 合格**：无循环（`memcpy` 由编译器内联为定长指令序列）。
**D3 合格**：满/空/不稳定均非阻塞返回。

### 3. 逐项发现与修复详情

#### F-1 ｜ 低（注释缺失）｜ mempool alloc 临界区 IRQ-off 窗口随 obj_size 增大

- **位置**：`Source/core/bm_mempool.c`，`bm_mempool_alloc` 内 `memset` 调用处（原约第 173 行）
- **现象**：`memset(obj, 0, pool->obj_size)` 在 `BM_CRITICAL` 关中断窗口内执行，时长 ∝ obj_size，但原注释未提示硬实时剖面须登记此窗口。
- **修复内容**：在现有"清零在锁内完成"注释块内追加 `[F-1 IRQ-off 窗口]` 段落，明确"BM_CRITICAL 关中断的最坏时长 ∝ 最大池对象 obj_size，会抬高全局最坏中断延迟；硬实时剖面须对大对象池的 obj_size 设上限并据此登记最坏 IRQ-off 窗口"。
- **行为变更**：无。纯注释。
- **残留风险**：用户未阅读注释仍可能漏登记，但框架无法强制编译期校验 obj_size 上限（属部署约束，需移植方承担）。

#### F-2 ｜ 低（注释缺失）｜ free 阻塞自旋 WCET 公式未含 obj_size 清零项

- **位置**：`Source/core/bm_mempool.c`，`mempool_lock_blocking` 函数注释（原约第 47 行）
- **现象**：原注释写"WCET = N_cores × 单次临界区时长"，但未指明单次临界区已含 alloc 侧的 `memset(obj_size)` 清零，使 WCET 公式不完整。
- **修复内容**：在注释块追加 `[F-2 WCET 补全]` 段落，明确"单次临界区含 alloc 侧 memset(obj_size)，故 free 最坏阻塞 WCET = N_cores × (位图操作 + obj_size 清零)"。
- **行为变更**：无。纯注释。
- **残留风险**：同 F-1，属部署约束文档层面。

#### F-3 ｜ 低（注释缺失）｜ alloc 扫描循环 WCET 须按满扫描取值，注释未说明

- **位置**：`Source/core/bm_mempool.c`，`bm_mempool_alloc` 外层 `for` 循环（原约第 154 行）
- **现象**：首适配线性扫描在池满/空槽末位时 WCET = O(count)，但原代码无注释，RTA 工程师可能误用平均情况估算。

#### F-4 ｜ 中（性能缺陷）｜ bm_mp enforce 自旋循环每轮执行 u64 除法

### 4. F-4 等价性推导

设：
- `freq`：定时器频率（ticks/秒，`uint32_t`，非 0）
- `period_us`：期望周期（微秒，`uint32_t`，非 0）
- `elapsed_ticks`：经过 ticks 数（`uint32_t` 回绕差）

**原判据**（us 域）：
```
elapsed_us = floor(elapsed_ticks * 1_000_000 / freq)
条件成立 ⟺ elapsed_us >= period_us
```

**等价变换**（乘法域）：

由 floor 除法定义：`floor(A/B) >= C` ⟺ `A >= C * B`（C、B 为正整数）

代入：
```
floor(elapsed_ticks * 1_000_000 / freq) >= period_us
⟺ elapsed_ticks * 1_000_000 >= period_us * freq
```

因此：
- **无舍入差异**：等式两边均为整数，floor 除法下严格等价，不引入 ceil/floor 分歧。
- **回绕安全**：`elapsed_ticks = (uint32_t)(now - start)` 使用 uint32_t 减法，单周期内回绕正确（与原函数实现一致）。
- **溢出分析**：`elapsed_ticks * 1_000_000` 在 `elapsed_ticks` 最大约 `4294` 秒对应 `4294 * freq` 时可能 u64 溢出，但实际 `MAX_SPINS = 100000`、`period_us` 通常 <= 10^6，`freq` 通常 <= 10^9，乘积 <= 10^15，远低于 u64 上界（~1.84×10^19），安全。

### 5. 审查局限

1. **纯静态审查**：未在硬件或 QEMU 上实测时序；F-4 的实际节省 cycle 数依目标 MCU 流水线（通常 ARM Cortex-M 软件 u64 除法 20-40 周期 vs u64 乘法 4-8 周期）。
2. **native_sim 不验时序**：本机构建工具链缺失（ESP-IDF/idf.py 未安装），无法编译运行单元测试；F-4 行为等价性由数学推导保证，非运行时验证。已对改动文件做语法级 sanity（字面检查，无新的 C 类型错误引入）。
3. **覆盖范围**：本次仅覆盖 `core/mempool`、`mp/bm_mp`、`mp/bm_mp_ipc_backend`、`core/bm_atomic`/`bm_critical`；`components/` 算法层 WCET、Demo、portable/arch 行为未在此次范围内。
