# 05 确定性验证操作手册（SOP）

> 配套设计见 confidential 设计 spec（私有 `docs/superpowers/specs/`）。
> 本手册是**可复现操作步骤**，供第三方照此重跑。
> 随能力分层累积：本版含 **L1（功能确定性，native）**；L2（QEMU icount）/L3（真机+LA）后续补。
>
> **认识论声明（必读）**：实测证据是"功能确定性 pass/fail + 与静态证明一致"的经验佐证，
> 不等于 WCET 证明。确定性的证明在静态构造（D1–D6 审查报告）。
> 本手册产出：L1 功能确定性 pass/fail。

---

## L1 功能确定性（native）

### 前置

| 条件 | 说明 |
|---|---|
| 工具链 | MSVC（Hoverboard 主路径）：先执行 `C:\MyApp\vcvars64.bat` |
| 构建命令 | `cmake -G Ninja -DBM_BUILD_TESTS=ON -DBM_ENABLE_BUS=ON -DBM_ENABLE_HRT=ON -DBM_ENABLE_ALGORITHM=ON <src> -B <build>` |
| 注意 | frameworks 副本改完须同步到 Hoverboard 副本后方可实跑 |

### 步骤

1. **A1+A2 可复现性（算法层）**

   ```
   ctest --test-dir <build> -R test_det_repro -V
   ```
   预期：`test_det_repro_clarke` / `test_det_repro_clarke_2shunt`（纯函数）、
   `test_det_repro_lpf1_state_trajectory`（有状态轨迹）全 PASS。

2. **A3 框架 round-trip 可复现性**

   ```
   ctest --test-dir <build> -R test_det_roundtrip -V
   ```
   预期：`test_det_repro_bus_latest_roundtrip`、`test_det_repro_event_dispatch_order` 全 PASS。

3. **B 零分配**

   ```
   ctest --test-dir <build> -R test_det_zeroalloc -V
   ```
   - GNU（权威）：陷阱分配器未触发即 PASS；报告话术："全套用例在 abort-on-alloc 分配器下跑到结束"。
   - MSVC（smoke）：CRT 钩子计数==0 即 PASS（钩子过滤 `_CRT_BLOCK`，仅计客户区分配）。

4. **C3+C4+C5 有界+非阻塞**

   ```
   ctest --test-dir <build> -R test_det_bounded -V
   ```
   预期：event max_events 有界 / mempool O(count) 有界 / atomic 饱和 全 PASS。

5. **C1 bus retry（DRY，`test_bus` 复用）**

   ```
   ctest --test-dir <build> -R test_bus -V
   ```
   预期：含 `test_latest_read_retry_bounded_under_contention` 的 `test_bus` 全 PASS。

6. **全 L1 标签一次跑**

   ```
   ctest --test-dir <build> -L determinism -V
   ```
   预期：`test_det_repro` / `test_det_roundtrip` / `test_det_zeroalloc` / `test_det_bounded` /
   `test_bus` 全选中且全 PASS。

### 结果记录（填入证据报告 L1 节）

| 项 | 命令 | 结果 | toolchain/配置 |
|---|---|---|---|
| A1+A2 可复现性 | `ctest -R test_det_repro` | PASS/FAIL | 填 MSVC/GNU + 版本 |
| A3 框架 round-trip | `ctest -R test_det_roundtrip` | PASS/FAIL | 填 |
| B 零分配 | `ctest -R test_det_zeroalloc` | PASS/FAIL | 填 |
| C3+C4+C5 有界 | `ctest -R test_det_bounded` | PASS/FAIL | 填 |
| C1 bus retry | `ctest -R test_bus` | PASS/FAIL | 填 |

> **可复现性边界**：仅同二进制同平台 run-to-run；跨平台逐位相同为非目标（x86 SSE vs ARM VFP）。
> 报告须在首页声明此边界，避免"正确地失败"（spec §3.5）。

## L2 QEMU icount 控制流确定性（ARM Cortex-A15 线）

> **认识论边界（必读）**：QEMU `-icount` 下周期数 ≠ 真硅片周期（无流水线/cache 模型）。
> L2 证明的是**控制流有界与可复现（Δ==0）**，不是 WCET。WCET 真值见 L3（真机 + 正点原子 LA）。
>
> **合法话术**："同相位两遍 Δ周期==0，控制流无数据依赖越界分支" ——可对外。
> **禁止话术**："QEMU 周期数≤预算 = 证明了 WCET" ——不可对外。

### 探针选型（关键，避免空断言）

ARM 周期源用 **PMU PMCCNTR**（CP15 c9,c13,0），配 `bm_probe_init()` 使能序列
（PMCR.E/C、D=0 每周期计数、PMCNTENSET.C）。**不用 CNTVCT**：实测 CNTVCT 在
`-icount` 下按翻译块（TB）边界记账，读数不随执行指令变化（注入 65 条 nop 仍 Δ==0），
会把 Δ==0 退化为恒真的空断言。PMCCNTR 随执行确定性推进，才能让 Δ==0 成为真证据。

### 前置

| 条件 | 说明 |
|---|---|
| 工具链 | `arm-none-eabi-gcc 13.3 rel1`（`C:\MyApp\Arm GNU Toolchain arm-none-eabi\13.3 rel1\bin`）|
| QEMU | `C:\Program Files\qemu\qemu-system-arm` ≥ 6.0（实测 11.0.50）|
| 构建目录 | `build_icount_arm`（在 `frameworks/Bmelod/` 根下） |
| 内存 | QEMU_ARGS 用 `-m 128M`，须与 `linker_smp.ld`（RAM 128M，`_estack`）一致；`-m 64M` 会栈越界 Data Abort |

### 步骤

1. **配置构建（仅首次或 CMakeLists 改变后）**

   ```powershell
   $env:PATH = "C:\MyApp\Arm GNU Toolchain arm-none-eabi\13.3 rel1\bin;" + $env:PATH
   cmake -B build_icount_arm -S tests/qemu_cortexa_icount -G Ninja `
       -DCMAKE_TOOLCHAIN_FILE=cmake/toolchain-arm-none-eabi-a.cmake
   ```

2. **构建并运行**

   ```powershell
   cmake --build build_icount_arm
   ctest --test-dir build_icount_arm -L determinism-l2 --output-on-failure -V
   ```

3. **读取串口输出（周期基线数字）**

   ```powershell
   Get-Content "build_icount_arm\det_icount_arm_serial.txt"
   ```

   - `# clarke_cyc1=N cyc2=N delta=0`：纯函数控制流确定。
   - `# bus_latest_cyc1=N cyc2=N delta=0`：bus LATEST 同相位 round-trip 控制流确定。

4. **漂移门比对（跨 commit 回归）**

   将当前 `cyc1` 与 `docs/superpowers/baselines/det_l2_arm_baseline.json`（私有）中的基线对比，
   偏差 >5% 即需人工 re-bless 基线并记录原因（spec §5.3 闸 c，不自动 FAIL）。

### 结果解读

| TAP 输出 | 含义 |
|---|---|
| `ok N - arm:probe_monotonic` | PMCCNTR 在 icount 下单调推进（前置条件） |
| `ok N - arm:clarke_delta_zero` | clarke 纯函数控制流确定（Δ==0） |
| `ok N - arm:bus_latest_delta_zero` | bus LATEST 同相位 round-trip 控制流确定（Δ==0） |
| `not ok N - arm:*_delta_zero` | 控制流非确定（icount 配置问题或真正非确定分支） |

> **bus 测法说明**：bus LATEST 是双缓冲+seqlock 的 **2 相状态机**，每 commit 交替 slot（周期=2）。
> 测量取**同相位**两遍（预热定相 + 测第一遍 + 丢弃一遍推进相位 + 测第二遍），否则比较的是
> 两个不同相位（slot 选择开销不同），Δ 会稳定约 7 周期——那是双缓冲固有结构，非控制流非确定。
