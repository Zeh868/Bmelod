# 01 框架能力演进路线规划（安全底座先行，智能上层后置）

> **状态：路线规划稿（未实现，待审查）**。本文把已讨论的若干候选能力与 **AI 推理引擎** 一并
> 纳入一张带优先级与依赖关系的演进路线图，用于后续排期与逐项立项。每项均给出 **进框架/不进框架**
> 的边界、与现有资产的复用关系、以及可 native 验收的实施切片。本文不落地代码，供审查后逐项转入实现。

## 1. 目标与定位

**本文是什么：** Bmelod 从「确定性实时控制框架」向「确定性实时 + 端侧智能框架」演进的能力地图与推进顺序。

**本文不是什么：** 不是某一项的详细设计（详细设计各自另起文档，如 [06-固件升级](../06-固件升级/01-A-B-swap升级方案设计.md)）；不是承诺，是排期依据。

**贯穿红线（所有能力共同遵守）：**

- 零动态分配、有界 WCET、ISR 不可阻塞、core/mp 热路径禁浮点——既有铁律不为任何新能力让步。
- 新能力优先落 **component 层**（当前 0.1 骨架），尽量**复用既有积木**而非另造底座。
- 平台相关部分走 HAL/后端抽象，保持 native 可测。

## 2. 总体策略：先把「安全实时」招牌做硬，再加「智能」上层

一句话主线：

> **安全底座（梯队 A）→ 通信与隔离扩展（梯队 B）→ 工程基建（梯队 C）→ AI 智能上层（梯队 D）。**

AI 推理的差异化价值——「**能证明在跑 AI 的同时不丢实时性**」——的前提，恰恰是先有运行时 deadline 监控（SAFE-2）和故障降级（SAFE-1）。**顺序不可反**：没有安全底座的 AI，是「实时性被 AI 拖垮的框架」，比不加更糟。

## 3. 能力地图与优先级总览

| 代号 | 能力 | 梯队 | 进框架定位 | 关键前置 |
|---|---|---|---|---|

| **SAFE-2** | 运行时 deadline / WCET 超额监控 | A 安全底座 | mp + component | — |
| **SAFE-3** | 传感器合理性校验 + 多路 lite 表决 | A 安全底座 | component | — |
| **OBS-1** | 黑匣子 / flight recorder（事件追踪环 + 崩溃冻结 + 回放） | A 安全底座 | component | TIME-1（多核时间戳对齐，可后补） |

## 4. 梯队 A：安全底座（最高优先，AI 前置）

### SAFE-1 FDIR 故障管理 + safe-state 降级编排

- **价值/痛点：** 现状是零散 fail-fast，缺统一的「故障检测→隔离→恢复→安全降级」状态机。平衡车「复位后 I2C 卡死、只能断电」即缺这一层的典型表现。
- **进框架：** 故障源注册、健康监控聚合、分级降级状态机（正常/降级/safe-state）、恢复策略钩子。
- **不进框架：** 具体故障判据与 safe-state 物理动作（断电/抱闸/限速）由应用提供回调。
- **复用：** `bm_wdg` + `bm_event` + 既有 `BM_ERR_*` 错误码。

### SAFE-2 运行时 deadline / WCET 超额监控

- **价值/痛点：** 有**静态** RTA 预算，但运行时无人盯「实际是否超预算」。补一层执行时间预算监控，超时即 fail-fast，把静态证明与运行时落地闭环。**这是 AI-1 隔离论证的硬前置。**
- **进框架：** 每任务/每段预算登记、运行时计时、deadline overrun 检测与回调（联动 SAFE-1）。
- **复用：** `bm_uptime` 单调时钟 + `bm_mp` 调度。

### SAFE-3 传感器合理性校验 + 多路 lite 表决

- **价值：** range / rate / cross-check + 轻量表决，建立「输入可信度」层；STALE 容错已有，差系统化封装。
- **进框架：** 合理性校验原语、N 取 M lite 表决、可信度标记随数据流转。
- **复用：** `bm_bus` LATEST + 已落地的 STALE 容错。

### OBS-1 黑匣子 / flight recorder

- **价值：** 崩溃现场冻结 + 离线回放，对「真机难复现」（I2C 卡死、timing flaky）场景价值极高。
- **进框架：** 定长事件追踪环、关键现场快照、导出/回放格式。
- **复用：** `bm_event` + `bm_stream` + `bm_mempool` 环形缓冲。
- **依赖：** 多核时间戳对齐需 TIME-1（单核可先行，多核回放再补）。

## 5. 梯队 B：通信与隔离扩展

- **COMM-1 CAN / CAN-FD：** 有强大的*内部* bus，缺*对外*确定性现场总线；电机/车类应用刚需，是走向分布式实时的关键一跳。进框架=协议栈与帧调度（平台无关），不进=具体 CAN 控制器驱动（走 HAL 后端）。
- **ISO-1 MPU 内存隔离：** 组件/任务隔离 + 栈溢出保护；确定性框架配 MPU 是自然延伸，也是 SIL 准备硬指标。进框架=MPU 区域抽象与配置，不进=各 MCU 的 MPU 寄存器细节（走 port）。
- **TIME-1 跨核时间同步：** 单核 uptime 已有，多核需核间时间基对齐（PTP-lite），否则 OBS-1 黑匣子、AI-1 结果时间戳跨核对不齐。**OBS-1 与 AI-1 的公共依赖。**
- **CTRL-1 状态估计组件化：** 把平衡车已有的姿态估计（卡尔曼/互补滤波）抽成**通用 component**，填充 component 层 0.1 骨架，避免每个应用重写。

## 6. 梯队 C：工程基建

- **BOOT-1 安全启动 + 镜像签名：** 与已成稿的 [A/B 固件升级设计](../06-固件升级/01-A-B-swap升级方案设计.md) 是一对——升级不验签名等于敞口；该设计已为签名预留扩展位，建议捆绑推进。
- **OBS-2 jitter / CPU 负载度量：** 实测最坏延迟、抖动统计，**与 RTA 静态预算对账**，把「我证明过」变成「我持续验证」。

## 7. 梯队 D：AI 推理引擎（AI-1）

> 详细架构另起专文；本节定纲与边界。

**定位：隔离的软实时组件，绝不进硬实时控制通路。**

**数据流：**

```
传感器 → bus(stream/LATEST) → 预处理 → bm_inference(独占核, 软实时, 静态 arena)
                                              ↓ 结果(带时间戳/置信度)
控制核(HRT) ← bus LATEST(seqlock, 有界自旋读) ← 推理结果   （读不到/过期 → STALE 降级，绝不等）
```

- **进框架：** `bm_inference` 组件 + 可插拔 backend 接口（复用 IPC/bus 已验证的 vtable + 「bounded/non-blocking」契约登记）、静态张量 arena（复用 `bm_mempool`）、结果经 bus LATEST 异步交付。
- **不进框架：** 具体推理引擎实现与模型（走 backend）、厂商 NPU/加速器寄存器细节（走 HAL accelerator 后端）。
- **引擎选型：** MCU 首选 **CMSIS-NN / TFLite-Micro**（静态 arena、无堆、int8 无浮点、Apache-2.0 可并入 GPLv3）；ESP32 可选 ESP-DL。**不上** ONNX Runtime full / 通用动态图引擎（动态分配 + WCET 不可预测，与框架根本冲突）。
- **确定性论点：** 固定网络 + 固定输入尺寸 ⇒ 推理本身 WCET 有界、可纳入 RTA；隔离核 + SAFE-2 deadline 监控共同**证明控制核 WCET 不受推理影响**——这是别人抄不走的卖点。
- **硬约束：** 单次推理远超控制环周期 ⇒ 必须异步解耦；int8 优先，浮点只允许在推理核软实时上下文，永不进 HRT ISR；arena 内存预算先算清纳入静态分配表。
- **依赖：** **SAFE-1（降级）、SAFE-2（deadline 证明）、TIME-1（结果时间戳）** 必须先就位。

## 8. 依赖关系与推进顺序

```
梯队A 安全底座 ──┬─ SAFE-1 FDIR ───────────┐
                 ├─ SAFE-2 deadline 监控 ───┼──► AI-1 推理引擎（梯队D）
                 ├─ SAFE-3 传感器表决       │
                 └─ OBS-1 黑匣子 ──┐        │
                                   │        │
梯队B ── TIME-1 跨核时间同步 ──────┴────────┘   （OBS-1 多核回放 / AI-1 时间戳的公共前置）
        COMM-1 CAN / ISO-1 MPU / CTRL-1 状态估计   （相对独立，可并行）
梯队C ── BOOT-1 安全启动 ◄── 配 06 A/B 升级设计
        OBS-2 jitter 度量 ◄── 复用 SAFE-2 计时基础设施
```

**关键结论：AI-1 排在 SAFE-1 / SAFE-2 / TIME-1 之后。** 安全底座未就位前不启动 AI 推理落地。

## 9. 分阶段路线图（建议排期）

| 阶段 | 内容 | 验收 |
|---|---|---|
| **P1 安全底座** | SAFE-2 deadline 监控 → SAFE-1 FDIR → SAFE-3 表决 → OBS-1 黑匣子（单核） | native ctest，每项独立用例 |
| **P2 隔离与时间** | TIME-1 跨核时间同步 → ISO-1 MPU；并行 CTRL-1 状态估计组件化 | native + QEMU SMP（时间对齐、隔离） |
| **P3 通信与基建** | COMM-1 CAN/CAN-FD → BOOT-1 安全启动签名（配 06）→ OBS-2 度量对账 | native（协议/签名）+ 上板（CAN PHY） |
| **P4 智能上层** | AI-1：组件骨架 + CMSIS-NN backend（最小固定网络）→ 多核隔离落地 → 真机/NPU backend | native 数据流 → **QEMU SMP 证明推理满载不影响控制核 WCET** → 上板 |

每阶段内部再细分 native 可测切片，立项时各自起详细设计文档。

## 10. 风险与红线

- **顺序红线：** AI-1 不得先于安全底座落地（见 §2/§8）。
- **隔离红线：** 推理一旦同步进入 HRT 控制环，§7 的核心卖点立即崩塌——任何「图省事同步调用」的改动一票否决。
- **内存预算：** AI arena、黑匣子环、CAN 缓冲均须先纳入静态分配表，确认目标芯片 RAM 放得下。
- **浮点污染：** NN/估计器的浮点不得渗入 core/mp 热路径与 ISR。
- **范围蔓延：** MPC/TSN/抢占内核明确不进框架核心（§3），按应用侧单独引入。

## 11. 待决策项（审查时确认）

1. **优先级确认：** 是否认可「安全底座先行、AI 后置」的总顺序，及 AI-1 排在 P4。
2. **首批立项：** P1 内 SAFE-2 / SAFE-1 / SAFE-3 / OBS-1 的先后与是否同批。
3. **目标平台权重：** COMM-1（CAN）面向裸机 MCU；若近期重心在 ESP32（灯哥 V4），其 OTA/通信优先级需相应调整。
4. **AI 引擎选型锁定：** CMSIS-NN 优先是否拍板，ESP-DL 是否作为 ESP32 分支。
5. **推进粒度：** 各能力一次做完 vs 按 native 切片分批审查（沿用 06 的切片风格）。
6. **范围边界确认：** MPC/TSN/抢占内核「不进框架」是否最终确认。
